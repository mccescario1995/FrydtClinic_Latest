# Plan: Refactor SMS API to Semaphore Provider

## Overview
Refactor the existing SMS API implementation from iProgSMS to Semaphore. This involves updating API endpoints, request/response handling, OTP verification logic, and configuration settings while maintaining existing functionality for outgoing SMS and OTP features.

## Current Implementation Analysis
- **Provider**: iProgSMS
- **Key Components**:
  - `SmsService.php`: Main service handling SMS/OTP operations
  - `SmsLog.php`: Logging model for all SMS transactions
  - `User.php`: OTP fields and methods
  - Database settings: `iprogsms_token`, `iprogsms_url`
- **API Endpoints**:
  - SMS: `POST https://www.iprogsms.com/api/v1/sms_messages`
  - OTP Send: `POST https://www.iprogsms.com/api/v1/otp/send_otp`
  - OTP Verify: `POST https://www.iprogsms.com/api/v1/otp/verify_otp`
  - Credits: `GET https://www.iprogsms.com/api/v1/account/sms_credits`

## Proposed Changes

### 1. Configuration Updates
- Add new environment variable: `SEMAPHORE_API_KEY` in `.env` file (user will provide the API key)
- Update config files to use `env('SEMAPHORE_API_KEY')` for authentication
- Update base URL to `https://api.semaphore.co/api/v4`
- Update admin settings UI (`resources/views/admin-portal/settings.blade.php`) to remove API key management since it's now in .env
- Remove iProgSMS-specific configurations from database settings

### 1.5. UI Enhancements
- Add rate limiting protection to the Check Credits button:
  - Disable button immediately after click
  - Display countdown timer showing remaining cooldown (30 seconds based on 2 calls/minute rate limit)
  - Re-enable button when cooldown expires
  - Show appropriate loading states and user feedback

### 2. SmsService.php Refactoring
- **API Client Updates**:
  - Change base URL to `https://api.semaphore.co/api/v4`
  - Update authentication from `api_token` to `apikey`
  - Change parameter names: `phone_number` → `number`
  - Update sender name handling (current: "kaprets", new: default "SEMAPHORE")

- **SMS Sending (`sendSms` method)**:
  - Update endpoint to `/messages`
  - Handle response fields: `message_id`, `status` (map to existing log statuses)
  - Status mapping:
    - "Queued" → "queued"
    - "Pending" → "pending" 
    - "Sent" → "sent"
    - "Failed" → "failed"
    - "Refunded" → "refunded"
  - Implement rate limiting (120 calls/minute) using Laravel's throttle middleware or custom logic

- **OTP Sending (`sendOtp` method)**:
  - Update endpoint to `/otp`
  - Use `{otp}` placeholder in message template
  - Store returned `code` in user model (`otp_code` field)
  - Handle auto-generated vs custom codes (prefer auto-generated for consistency)

- **OTP Verification (`verifyOtp` method)**:
  - Change from API verification to local verification
  - Compare user input against stored `otp_code`
  - Maintain existing attempt limits and expiry logic
  - Remove API call to verify endpoint

- **Credits Check (`getCredits` method)**:
  - Update to use new endpoint: `GET /api/v4/account`
  - Implement rate limiting (2 calls/minute)
  - Parse `credit_balance` from response
  - Return credit balance for display in admin panel

### 3. Database and Model Updates
- **SmsLog Model**:
  - Update status field to accommodate new status values
  - Ensure `message_id` from Semaphore is stored
  - Remove any iProgSMS-specific fields if present

- **User Model**:
  - Ensure OTP fields remain compatible
  - No changes needed for local verification

### 4. Phone Number Formatting
- Verify compatibility: current formats to `63xxxxxxxxx`, confirm Semaphore accepts this format
- Test with various Philippine number formats (with/without +63, with/without 0 prefix)

### 5. Error Handling and Logging
- Update error responses to handle Semaphore's JSON response format
- Maintain comprehensive logging in SmsLog table
- Add specific error handling for rate limiting (HTTP 429)

### 6. Testing and Validation
- Test all SMS sending methods (appointment bookings, confirmations, etc.)
- Test OTP generation and verification flow
- Validate phone number formatting across different inputs
- Test rate limiting behavior for SMS/OTP and credits check
- Test Check Credits button rate limiting UI (disabled state, countdown timer)
- Verify admin panel SMS logs display correctly with new statuses
- Verify credits display updates correctly after successful API calls

### 7. Migration Steps
1. Backup current configuration and logs
2. Add `SEMAPHORE_API_KEY` to `.env` file
3. Deploy code changes
4. Test SMS sending in staging environment
5. Monitor logs for any issues
6. Update documentation

## Potential Challenges
- **Rate Limiting**: Implement 120 calls/minute for SMS/OTP and 2 calls/minute for account checks
- **Status Mapping**: Ensure all Semaphore statuses map appropriately to existing log statuses
- **Environment Configuration**: Ensure .env variable is properly loaded and secured
