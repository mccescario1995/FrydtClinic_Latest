# Phone Number Setup Issue Investigation and Fix

## Problem Summary
Users are being redirected to set-phone page instead of verify-otp page despite having phone numbers in the database. The middleware fails to load patientProfile relationships, causing $user->patientProfile to be null, which makes the phone check fail.

## Root Cause Analysis
1. **Missing relationship loading**: `CheckPinVerification.php` gets user via `backpack_auth()->user()` but doesn't load `patientProfile`/`employeeProfile` relationships
2. **Debug evidence**: User object shows `#with: []` and `#relations: []` - no relationships loaded
3. **Logic flaw**: When `$user->patientProfile` is null, `$phone` becomes null, triggering set-phone redirect instead of verify-otp redirect

## Investigation Findings
- Patient profiles exist in database with phone numbers (debug shows "639976747550")
- The middleware `getUserPhone()` method tries to access `$user->patientProfile->phone` but relationship isn't loaded
- This causes `$phone` to be null, triggering the wrong redirect (set-phone instead of verify-otp)
- OTP verification logic is correct but never reached due to relationship loading issue

## Proposed Solution
1. **Fix relationship loading**: Load patientProfile and employeeProfile relationships in the middleware
2. **Ensure correct redirect logic**: Users with phones should go to verify-otp, users without phones should go to set-phone

## Implementation Steps
1. Modify `CheckPinVerification.php`: Change `$user = backpack_auth()->user();` to `$user = backpack_auth()->user()->load(['patientProfile', 'employeeProfile']);`
2. Test that users with existing phone numbers are redirected to verify-otp page
3. Test that users without phone numbers are redirected to set-phone page

## Files to Modify
- `app/Http/Middleware/CheckPinVerification.php` (add relationship loading)

## Testing Requirements
- Test patient with phone + expired OTP → should redirect to verify-otp
- Test patient without phone → should redirect to set-phone
- Test employee with phone + expired OTP → should redirect to verify-otp
- Test employee without phone → should redirect to set-phone

## Additional Issue: Logout Redirect Loop

## Problem Summary
After implementing the phone/OTP verification fix, users are experiencing a redirect loop when trying to logout from the set-phone page. After clicking logout, they are redirected back to the set-phone page instead of successfully logging out.

## Root Cause Analysis
The issue occurs because after logout, when the user is redirected to the homepage, the CheckPinVerification middleware is still detecting the user as authenticated due to improper session clearing or guard usage in the logout process.

## Investigation Findings
From examining the LoginController.php logout method:
```php
public function logout(Request $request)
{
    $user = backpack_auth()->user();
    $this->guard()->logout();
    $request->session()->invalidate();
    $request->session()->regenerateToken();
    // ... clear OTP fields
    return redirect()->route('homepage');
}
```

The potential issue is the order of operations and which guard is being used for logout vs. authentication checking.

## Proposed Solution
Fix the logout process to properly clear authentication state by:
1. Using the same guard for checking and logging out
2. Clearing session before invalidating it
3. Ensuring proper session flush

## Implementation Steps for Logout Fix
1. Modify LoginController.php logout method to use backpack guard explicitly
2. Change order: invalidate session before logout
3. Add session flush for good measure
4. Test logout flow thoroughly

- `app/Http/Controllers\Auth\LoginController.php` (logout method)
